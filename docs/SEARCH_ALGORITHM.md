## Overview

The algorithm answers the question:

> *"What fusion chains produce **Persona X** such that it ends up with skills **SвӮҒ, SвӮӮ, вҖҰ, SвӮҷ**?"*

All search logic runs inside a **Web Worker** (`fusionSearch.worker.js`) to keep the UI responsive. The worker imports `FusionCalculator.js` directly, performing recipe-map precomputation once at worker init.

## Phase 1 вҖ” Precomputation

All precomputation runs **once** at module-load time. The cost is amortised across all user queries.

### 1.1 Arcana Grouping

Every Persona is grouped into a map keyed by its arcana (called `race` in the data). Each group is sorted ascending by base level. This is required for the fusion-result lookup described below.

### 1.2 Normal Fusion Result (`getNormalFusionResult`)

Given two Personas **A** and **B**:

1. Look up the **result arcana** from the lower-triangular fusion chart using both Personas' arcanas.
2. Compute the **average level**: `avg = (lvlA + lvlB) / 2`.
   - **Same arcana** вҶ’ find the **highest-level Persona вүӨ avg + 1**, excluding A, B, and specials. Verified in-game: Abaddon(76,Devil) Г— Baphomet(30,Devil) вҶ’ Succubus(47) [avg=53, cap=54, highest Devil вүӨ 54].
   - **Different arcana** вҶ’ `target = Math.floor(avg) + 1`; find the **lowest-level non-special Persona вүҘ target**, falling back to the highest-level non-special if none exist. Verified in-game: Alice(68,Death) Г— Kaiwan(42,Star) вҶ’ Horus(67) [avg=55, target=56, first Sun вүҘ 56]; Nebiros(52,Hermit) Г— Trumpeter(59,Judgement) вҶ’ Byakko(63) [avg=55.5, target=56, first Temperance вүҘ 56].
3. Special-recipe Personas are always excluded from normal fusion results.

### 1.3 Recipe Map (`recipeMap`)

A reverse-lookup table is built:

```
recipeMap[resultPersonaName] = [
  { ingredients: ["PersonaA", "PersonaB"], isSpecial: false },
  { ingredients: ["PersonaC", "PersonaD", "PersonaE"], isSpecial: true },
  ...
]
```

- **Special recipes** are inserted directly from `special-recipes.json`.
- **Normal fusions** are generated by iterating all unique Persona pairs (O(nВІ) with ~300 Personas вүҲ ~45 000 pairs).

### 1.4 Recipe Pre-Sorting

After building the recipe map, each Persona's recipe list is sorted by the **maximum ingredient level** in ascending order. This ensures that during the depth-limited search, lower-level (and therefore more accessible) paths are explored and returned first.

## Phase 2 вҖ” Backward-Chaining Search

### Entry Point: `findFusionPaths(targetPersona, targetSkills, maxDepth, currentLevel, requiredPersonaCount)`

1. **Inheritance validation** вҖ” Before searching, each target skill is checked against the target Persona's inheritance bitmask. If any skill is uninheritable, an error is returned immediately.
2. **Recursive search** вҖ” Calls `searchTree(targetPersona, targetSkills, maxDepth, memo)`.

### Core Recursion: `searchTree(personaName, requiredSkills, maxDepth, memo)`

```
searchTree(persona, requiredSkills, depth):
    if memo hit вҶ’ return cached
    if persona can't inherit all requiredSkills вҶ’ return []
    
    innate вҶҗ skills the persona learns natively
    stillRequired вҶҗ requiredSkills - innate
    
    if stillRequired is empty вҶ’ return [leaf node]
    if depth = 0 вҶ’ return []
    
    for each recipe that produces persona:
        for each distribution of stillRequired across ingredients:
            for each ingredient:
                recurse with assigned skills at depth-1
            if all ingredients resolved вҶ’ record path
            
    memo[key] = results
    return results
```

### Skill Distribution (`distributeSkills`)

When a recipe has *k* ingredients and there are *m* skills still needed, the algorithm generates **all possible assignments** of skills to ingredients. Each skill must be sourced from exactly one ingredient.

This is computed recursively: for each skill, try placing it in each of the *k* buckets, producing `k^m` total distributions.

The algorithm takes the **first valid child path** per ingredient (not all combinations) to keep the search tractable.

### Memoisation

Results are cached with the key:

```
`${personaName}:${sortedSkillNames}:${maxDepth}`
```

This prevents redundant computation when the same sub-problem (same Persona, same required skill set, same remaining depth) is encountered through different parent paths.

### Path Limits

- **5 paths per sub-problem** вҖ” The search stops collecting valid paths for a given `(persona, skills, depth)` state after finding 5. This prevents memory explosion in the exponential search space.

## Phase 3 вҖ” Post-Search Filtering & Sorting

After `searchTree` returns all discovered paths:

### 3.1 Persona Count Filter (optional)

If the user specifies a required persona count, paths are filtered to include only those whose total recursive persona count matches exactly.

`getPathPersonaCount(path)` counts the root node plus all nodes in all sub-trees recursively.

### 3.2 Level-Aware Sorting

Paths are sorted using a two-tier comparator:

1. **Achievability** вҖ” Paths where `getPathMaxLevel(path) вүӨ currentLevel` are placed before paths that exceed the player's level.
2. **Level ascending** вҖ” Within each tier, paths are sorted by their maximum ingredient level, lowest first.

`getPathMaxLevel(path)` recursively finds the highest base level among all Personas in the fusion tree.

## Phase 4 вҖ” Web Worker Integration

The search no longer runs on the main thread. All depth iterations happen inside a dedicated Web Worker.

### Worker Lifecycle

- **Created once** at app mount in a `useEffect`, stored in `workerRef`.
- **Self-healing**: if the worker errors, `workerHealthyRef` is set to `false`; the next `handleCalculate` call recreates it.
- **Terminated** on component unmount.

### Message Protocol

| Direction | Type | Payload |
|-----------|------|---------|
| Main вҶ’ Worker | `search` | `{ targetPersona, targetSkills, requiredPersonas }` |
| Main вҶ’ Worker | `cancel` | _(none)_ |
| Worker вҶ’ Main | `progress` | `{ depth, paths }` вҖ” new unique paths found at this depth |
| Worker вҶ’ Main | `done` | _(none)_ вҖ” all depths exhausted |
| Worker вҶ’ Main | `error` | `{ message }` вҖ” unexpected error |

### Depth Iteration

The worker loops depth from 1 to `MAX_DEPTH` (20) calling `searchTree()` or `generateFusionTrees()` at each level. After each depth:

1. **Filter** by `requiredPersonas` if set.
2. **Deduplicate** against all previously seen paths using `getPathPersonaNames()` вҖ” a Set keyed by sorted persona names.
3. **Enrich** each new path with `_maxLevel` and `_nodeCount` for main-thread sorting.
4. **Post** `progress` message with only new unique paths.

The loop stops when two consecutive depths produce zero paths (exhaustion).

### Main-Thread Sort

Each `progress` message appends new paths to the state array. A `sortedPaths` useMemo re-sorts the entire list whenever `paths` or `currentLevel` changes:

```
sort priority:
  1. Achievable at current level (_maxLevel вүӨ currentLevel)
  2. Fewest nodes (_nodeCount)
  3. Lowest max level (_maxLevel)
```

### Cancel Behaviour

When the main thread needs to stop a running search (user clicks Cancel, changes params, navigates to Database view), it sends `{ type: 'cancel' }`. The worker checks a `cancelled` flag at the top of each depth iteration and exits cleanly.

## Key Data Structures

### Fusion Path Node

Each node in a returned fusion path has this shape:

```js
{
  persona: "Thanatos",          // The Persona at this node
  skillsProvided: ["Megidola"], // Skills this sub-tree is responsible for
  innateProvided: ["Megidola"], // Subset of skillsProvided that this Persona learns natively
  ingredients: [                // Child nodes (empty for leaf / base Personas)
    { persona: "Loa", ... },
    { persona: "Mot", ... }
  ]
}
```

### Recipe Map Entry

```js
{
  ingredients: ["Orpheus", "Apsaras"],  // Persona names needed
  isSpecial: false                       // true for multi-ingredient special fusions
}
```

## Complexity & Limits

| Dimension | Bound |
|---|---|---|
| Recipe map construction | O(nВІ) where n вүҲ 300 Personas (~45k pairs), runs once in each thread |
| Skill distribution per recipe | O(k^m) where k = ingredients, m = remaining skills |
| Search tree branching | Bounded by 5 paths/state Г— recipe count Г— distributions |
| Memoisation | Prevents re-expansion of identical `(persona, skills, depth)` states |
| Practical depth | Unlimited вҖ” worker iterates until two consecutive depths yield zero paths |
| Max depth safety limit | 20 (`MAX_DEPTH` in `fusionSearch.worker.js`) |

The pre-sorting of recipes by ingredient level is the primary heuristic that makes the first results returned the most useful ones, since lower-level paths are explored and recorded before higher-level alternatives.

### UI Impact

Because the search runs in a Web Worker:

- **No main-thread blocking** вҖ” the UI stays responsive even at high depths.
- **Progressive rendering** вҖ” paths appear as soon as each depth finishes.
- **No "See Deeper Paths" button** вҖ” the search continues until all possibilities are explored.
- **Path count cap removed** вҖ” all unique paths across all depths are shown.
